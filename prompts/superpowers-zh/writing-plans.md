# 写计划（Writing Plans）

假设工程师对你的代码库零上下文且品味堪忧，写全面的实现计划。

## 概述

记录他们需要知道的一切：每个任务涉及哪些文件、代码、测试、需要检查的文档、怎么测试。把整个计划给成小任务。

**假设：** 技能熟练的开发者，但对你的工具和问题域几乎一无所知。不太懂好的测试设计。

**保存计划到：** `docs/superpowers/plans/YYYY-MM-DD-<功能名>.md`

## 范围检查

如果规格覆盖多个独立子系统，建议拆成独立计划 — 每个子系统一个。每个计划应产出可工作、可测试的软件。

## 文件结构

定义任务前，先规划要创建或修改的文件及每个文件的职责。

- 设计边界清晰、接口定义良好的单元。每个文件有单一清晰职责。
- 你推理一次能放入上下文的代码效果最好，文件更聚焦时编辑更可靠。
- 一起变更的文件应该在一起。按职责分，不按技术层分。
- 在已有代码库中，遵循已有模式。如果代码库用大文件，不要单方面重构 — 但如果你修改的文件已经过大，包含拆分在计划中是合理的。

## 小任务粒度

**每个步骤一个动作（2-5 分钟）：**
- "写失败测试" — 步骤
- "运行确保失败" — 步骤
- "写最少代码通过" — 步骤
- "运行测试确保通过" — 步骤
- "提交" — 步骤

## 计划文档头部

```markdown
# [功能名] 实现计划

**目标：** 一句话描述要构建什么

**架构：** 2-3 句话描述方案

**技术栈：** 关键技术/库

---
```

## 任务结构

```markdown
### 任务 N：[组件名]

**文件：**
- 创建：`精确/路径/文件.py`
- 修改：`精确/路径/已有.py:123-145`
- 测试：`tests/精确/路径/test.py`

- [ ] **步骤 1：写失败测试**

```python
def test_具体行为():
    result = function(input)
    assert result == expected
```

- [ ] **步骤 2：运行测试验证失败**

运行：`pytest tests/path/test.py::test_name -v`
预期：FAIL with "function not defined"

- [ ] **步骤 3：写最少实现**

```python
def function(input):
    return expected
```

- [ ] **步骤 4：运行测试验证通过**

运行：`pytest tests/path/test.py::test_name -v`
预期：PASS

- [ ] **步骤 5：提交**

```bash
git add tests/path/test.py src/path/file.py
git commit -m "feat: add specific feature"
```
```

## 不允许占位符

每个步骤必须包含工程师需要的实际内容。这些是**计划失败** — 绝不写：
- "TBD"、"TODO"、"稍后实现"、"填写细节"
- "添加适当的错误处理" / "添加验证" / "处理边界情况"
- "为上述写测试"（没有实际测试代码）
- "类似任务 N"（重复代码 — 工程师可能乱序阅读任务）
- 描述做什么但没有怎么做（代码步骤必须有代码块）
- 引用任何任务中未定义的类型、函数或方法

## 记住

- 始终用精确文件路径
- 每个步骤有完整代码 — 如果步骤改代码，展示代码
- 精确命令和预期输出
- DRY、YAGNI、TDD、频繁提交

## 自审

写完完整计划后，用新眼光审视规格和计划：

**1. 规格覆盖：** 浏览规格的每个部分/需求。能指出哪个任务实现了它吗？列出缺口。

**2. 占位符扫描：** 搜索"不允许占位符"中的红线。修复它们。

**3. 类型一致性：** 后面任务中的类型、方法签名、属性名与前面定义的一致吗？任务 3 中叫 `clearLayers()` 但任务 7 中叫 `clearFullLayers()` 是 Bug。

发现任何问题直接修复。不需要重新审查。

## 执行交接

保存计划后，提供两种执行选项：

**1. 子代理驱动（推荐）** — 每个任务派发新子代理，任务间审查

**2. 内联执行** — 在当前会话中执行任务，批量检查点

# 系统化调试（Systematic Debugging）

在遇到任何 bug、测试失败或意外行为时使用，在提出修复方案之前。

## 铁律

```
没有根因调查，就不许修复
```

## 四个阶段

### 阶段 1：根因调查
1. **仔细读错误信息** — 堆栈跟踪、行号、错误码
2. **稳定复现** — 精确步骤，每次都能触发
3. **检查最近变更** — git diff、最近提交、新依赖
4. **收集证据** — 在组件边界加日志

### 阶段 2：模式分析
1. **找正常工作的例子** — 类似代码能正常运行
2. **对比正常 vs 异常** — 区别在哪？
3. **检查依赖** — 版本变化、API 变化
4. **验证假设** — 你以为的 X 真的是 X 吗？

### 阶段 3：假设与验证
1. **形成一个假设** — "我认为 X 导致 Y，因为 Z"
2. **设计最小测试** — 最小改动来证明/证伪
3. **测试** — 证据支持假设吗？
4. **如果不支持** — 形成新假设，不要乱改

### 阶段 4：实施修复
1. **写失败测试** — 复现 bug
2. **最小修复** — 一次改动，一个修复
3. **验证** — 测试通过，无回归

## 停止信号

- 3 次修复失败 → 质疑你的理解，而不是代码
- 时间压力 → 系统化比乱试更快
- "快速修复" → 快速修复制造新 bug

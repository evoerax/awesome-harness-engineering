# 系统化调试（Systematic Debugging）

随机修复浪费时间并制造新 Bug。快速补丁掩盖深层问题。

**核心原则：** 永远先找根因再尝试修复。症状修复是失败。

## 铁律

```
没有根因调查，就不许修复
```

如果还没完成阶段 1，就不能提出修复方案。

## 何时使用

用于任何技术问题：
- 测试失败
- 生产环境 Bug
- 意外行为
- 性能问题
- 构建失败
- 集成问题

**尤其在以下情况使用：**
- 时间压力下（紧急情况让人想猜）
- "就一个快速修复"看似明显
- 已经试了多个修复
- 之前的修复没用
- 你不完全理解问题

**不要跳过：**
- 问题看起来简单（简单 Bug 也有根因）
- 你很赶（赶工保证返工）
- 老板要立刻修好（系统化比乱试更快）

## 四个阶段

**必须按顺序完成每个阶段。**

### 阶段 1：根因调查

**在尝试任何修复之前：**

1. **仔细读错误信息**
   - 不要跳过错误或警告
   - 它们经常包含确切的解决方案
   - 完整读堆栈跟踪
   - 记下行号、文件路径、错误码

2. **稳定复现**
   - 能可靠触发吗？
   - 精确步骤是什么？
   - 每次都发生吗？
   - 如果不可复现 → 收集更多数据，不猜

3. **检查最近变更**
   - 什么变更可能导致这个？
   - Git diff、最近提交
   - 新依赖、配置变更
   - 环境差异

4. **在多组件系统中收集证据**

   **当系统有多个组件时（CI → 构建 → 签名，API → 服务 → 数据库）：**

   **在提出修复前，加诊断日志：**
   ```
   对每个组件边界：
     - 记录进入组件的数据
     - 记录离开组件的数据
     - 验证环境/配置传播
     - 检查每层状态

   运行一次收集证据，显示哪里断了
   然后分析证据识别失败组件
   然后调查那个具体组件
   ```

5. **追溯数据流**
   - 坏值从哪来的？
   - 谁用坏值调用了这里？
   - 继续向上追溯直到找到源头
   - 在源头修复，不在症状处修复

### 阶段 2：模式分析

**修复前找模式：**

1. **找正常工作的例子**
   - 定位同代码库中类似的正常代码
   - 什么正常工作但跟坏的类似？

2. **对比参考**
   - 如果在实现模式，完整读参考实现
   - 不要略读 — 每行都读
   - 完全理解模式再应用

3. **识别差异**
   - 正常和异常之间什么不同？
   - 列出每个差异，无论多小
   - 不要假设"这个不可能影响"

4. **理解依赖**
   - 需要哪些其他组件？
   - 什么设置、配置、环境？
   - 做了什么假设？

### 阶段 3：假设与验证

**科学方法：**

1. **形成单一假设**
   - 清晰陈述："我认为 X 是根因，因为 Y"
   - 写下来
   - 具体，不模糊

2. **最小测试**
   - 做最小可能的改动来验证假设
   - 一次一个变量
   - 不要同时修多个东西

3. **验证再继续**
   - 有用吗？是 → 阶段 4
   - 没用？形成新假设
   - 不要在上面加更多修复

4. **不知道时**
   - 说"我不理解 X"
   - 不要假装知道
   - 求助
   - 继续研究

### 阶段 4：实施修复

**修根因，不修症状：**

1. **创建失败测试用例**
   - 最小复现
   - 尽量自动化测试
   - 修复前必须有

2. **实施单一修复**
   - 解决识别的根因
   - 一次一个改动
   - 不要"顺便"改进
   - 不要捆绑重构

3. **验证修复**
   - 测试现在通过？
   - 没有破坏其他测试？
   - 问题实际解决了？

4. **如果修复没用**
   - 停下
   - 数：你试了几个修复？
   - < 3：回到阶段 1，用新信息重新分析
   - **≥ 3：停下，质疑架构（见步骤 5）**
   - 不要在没有架构讨论的情况下尝试第 4 次修复

5. **如果 3+ 次修复失败：质疑架构**

   **表明架构问题的模式：**
   - 每次修复都暴露新的共享状态/耦合/不同位置的问题
   - 修复需要"大规模重构"
   - 每次修复在其他地方制造新症状

   **停下质疑基本假设：**
   - 这个模式根本上合理吗？
   - 我们是不是"纯粹因为惯性在坚持"？
   - 应该重构架构还是继续修症状？

   **在尝试更多修复前与搭档讨论**

   这不是失败的假设 — 这是错误的架构。

## 红线 — 停下，遵循流程

如果你发现自己想：
- "快速修一下，之后再调查"
- "就试试改 X 看看有没有用"
- "加多个改动，跑测试"
- "跳过测试，我手动验证"
- "可能是 X，让我修那个"
- "我不完全理解但这可能有用"
- "问题是主要问题：[列出修复但没调查]"
- 在追溯数据流前提方案
- "再试一次修复"（已经试了 2+ 次）
- 每次修复在不同位置暴露新问题

**以上全部 = 停下。回到阶段 1。**

**如果 3+ 次修复失败：** 质疑架构（见阶段 4 步骤 5）

## 常见借口

| 借口 | 现实 |
|------|------|
| "问题简单，不需要流程" | 简单问题也有根因。流程对简单 Bug 很快。 |
| "紧急情况，没时间走流程" | 系统化调试比乱试更快。 |
| "先试这个，再调查" | 第一次修复设定模式。从一开始就做对。 |
| "修好后再写测试" | 没测试的修复不持久。先测证明有效。 |
| "同时修多个省时间" | 无法隔离哪个有用。制造新 Bug。 |
| "参考太长，我改编模式" | 部分理解保证 Bug。完整读它。 |
| "我看到问题了，让我修" | 看到症状 ≠ 理解根因。 |
| "再试一次修复"（2+ 次失败后） | 3+ 次失败 = 架构问题。质疑模式，不再修复。 |

## 快速参考

| 阶段 | 关键活动 | 成功标准 |
|------|---------|---------|
| **1. 根因** | 读错误、复现、检查变更、收集证据 | 理解什么和为什么 |
| **2. 模式** | 找正常例子、对比 | 识别差异 |
| **3. 假设** | 形成理论、最小测试 | 确认或新假设 |
| **4. 实施** | 创建测试、修复、验证 | Bug 解决、测试通过 |

## 实际影响

调试会话数据：
- 系统化方法：15-30 分钟修复
- 随机修复方法：2-3 小时乱试
- 首次修复率：95% vs 40%
- 引入新 Bug：接近零 vs 常见

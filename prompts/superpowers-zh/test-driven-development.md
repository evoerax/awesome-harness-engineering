# 测试驱动开发（TDD）

先写测试。看着它失败。写最少代码通过。

**核心原则：** 如果你没有看着测试失败，你就不知道它测了对的东西。

**违反字面规则就是违反精神规则。**

## 何时使用

**始终：**
- 新功能
- Bug 修复
- 重构
- 行为变更

**例外（问你的搭档）：**
- 一次性原型
- 生成的代码
- 配置文件

想"就这一次跳过 TDD"？停。那是自我合理化。

## 铁律

```
没有失败的测试，就不许写生产代码
```

先写代码后写测试？删掉。重来。

**没有例外：**
- 不要"保留参考"
- 不要写测试时"改编"
- 不要看它
- 删掉就是删掉

从测试开始全新实现。句号。

## 红-绿-重构

### 红（RED）— 写失败的测试

写一个最小测试，说明应该发生什么。

```typescript
// 好
test('重试失败操作 3 次', async () => {
  let attempts = 0;
  const operation = () => {
    attempts++;
    if (attempts < 3) throw new Error('fail');
    return 'success';
  };
  const result = await retryOperation(operation);
  expect(result).toBe('success');
  expect(attempts).toBe(3);
});

// 差
test('重试能用', async () => {
  const mock = jest.fn()
    .mockRejectedValueOnce(new Error())
    .mockRejectedValueOnce(new Error())
    .mockResolvedValueOnce('success');
  await retryOperation(mock);
  expect(mock).toHaveBeenCalledTimes(3);
});
```

**要求：**
- 一个行为
- 清晰的名称
- 真实代码（除非不可避免不用 mock）

### 验证红 — 看它失败

**强制。永远不能跳过。**

```bash
npm test path/to/test.test.ts
```

确认：
- 测试失败（不是报错）
- 失败信息符合预期
- 因功能缺失失败（不是拼写错误）

**测试通过？** 你在测已有行为。修测试。
**测试报错？** 修报错，重跑直到正确失败。

### 绿（GREEN）— 最少代码通过

写最简代码让测试通过。

```typescript
// 好 — 刚好够
async function retryOperation<T>(fn: () => Promise<T>): Promise<T> {
  for (let i = 0; i < 3; i++) {
    try { return await fn(); }
    catch (e) { if (i === 2) throw e; }
  }
  throw new Error('unreachable');
}

// 差 — 过度工程
async function retryOperation<T>(
  fn: () => Promise<T>,
  options?: { maxRetries?: number; backoff?: 'linear' | 'exponential'; onRetry?: (attempt: number) => void; }
): Promise<T> { /* YAGNI */ }
```

不加功能、不重构其他代码、不"改进"测试之外的东西。

### 验证绿 — 看它通过

**强制。**

```bash
npm test path/to/test.test.ts
```

确认：
- 测试通过
- 其他测试仍通过
- 输出干净（无错误、警告）

**测试失败？** 修代码，不修测试。
**其他测试失败？** 立即修复。

### 重构 — 清理

只有绿了之后：
- 消除重复
- 改善命名
- 提取辅助函数

保持测试绿。不加行为。

## 好测试

| 质量 | 好 | 差 |
|------|------|-----|
| **最小** | 一件事。名称有"和"？拆分。 | `test('验证邮箱和域名和空格')` |
| **清晰** | 名称描述行为 | `test('test1')` |
| **展示意图** | 展示期望 API | 隐藏代码应该做什么 |

## 常见借口

| 借口 | 现实 |
|------|------|
| "太简单不用测" | 简单代码也会坏。测试只要 30 秒。 |
| "我之后再测" | 测试立刻通过证明不了什么。 |
| "之后测能达到同样目的" | 之后测 = "这做了什么？" 先测 = "这应该做什么？" |
| "我已经手动测过" | 随机 ≠ 系统。没记录，不能重跑。 |
| "删 X 小时的工作太浪费" | 沉没成本谬误。保留未经验证的代码是技术债。 |
| "保留参考" | 你会改编它。那就是之后测。删掉就是删掉。 |
| "需要先探索" | 可以。扔掉探索成果，从 TDD 开始。 |
| "测试难写 = 设计不清晰" | 听测试的。难测 = 难用。 |
| "TDD 会拖慢速度" | TDD 比调试快。 |
| "手动测试更快" | 手动不证明边界情况。每次改动都要重测。 |

## 红线 — 停止，重来

- 先写代码后写测试
- 测试在实现之后
- 测试立刻通过
- 无法解释测试为什么失败
- 测试"之后"添加
- 合理化"就这一次"
- "我已经手动测过了"
- "保留参考"或"改编已有代码"
- "已经花 X 小时了，删掉太浪费"
- "TDD 是教条，我是务实的"
- "这次不一样因为..."

**以上任何一条出现 = 删掉代码，从 TDD 重来。**

## 卡住时

| 问题 | 解决方案 |
|------|---------|
| 不知道怎么测 | 写期望的 API。先写断言。问搭档。 |
| 测试太复杂 | 设计太复杂。简化接口。 |
| 必须 mock 一切 | 代码太耦合。用依赖注入。 |
| 测试设置庞大 | 提取辅助函数。还是复杂？简化设计。 |

## 验证清单

完成工作前检查：

- [ ] 每个新函数/方法都有测试
- [ ] 每个测试在实现前都看着失败
- [ ] 每个测试因预期原因失败（功能缺失，不是拼写错误）
- [ ] 为每个测试写了最少代码
- [ ] 所有测试通过
- [ ] 输出干净（无错误、警告）
- [ ] 测试用真实代码（除非不可避免不用 mock）
- [ ] 覆盖边界情况和错误

检查不了全部？你跳过了 TDD。重来。

## Bug 修复集成

发现 Bug？写失败测试复现它。遵循 TDD 循环。测试证明修复并防止回归。

永远不要在没有测试的情况下修复 Bug。

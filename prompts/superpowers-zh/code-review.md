# 代码审查（Code Review）

在完成任务、实现功能或合并前使用。

## 何时审查

**必须审查：**
- 每个任务完成后
- 主要功能完成时
- 合并到 main 前

**建议审查：**
- 卡住时（新视角）
- 重构前（基线检查）
- 修复复杂 bug 后

## 审查什么

1. **计划对齐** — 代码是否做了规格要求的？
2. **测试覆盖** — 边界情况测了吗？测试有意义吗？
3. **代码质量** — 命名清晰、函数小、无重复
4. **安全** — 无硬编码密钥、输入验证、错误处理
5. **性能** — 无明显 N+1、不必要的分配
6. **集成** — 会不会破坏现有功能？

## 审查流程

```
1. 先读计划/需求
2. 读 diff (git diff BASE..HEAD)
3. 检查测试通过
4. 检查问题：
   - 严重：必须修复才能继续
   - 重要：继续前修复
   - 轻微：记录后续处理
5. 带推理地报告发现
```

## 反模式

- 不理解计划就开始审查
- 纠结风格而漏掉逻辑错误
- 没真正读代码就批准
- "代码量小就不审查了"

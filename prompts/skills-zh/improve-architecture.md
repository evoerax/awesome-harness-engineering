# 改善代码库架构

探索代码库，找到架构摩擦点，提出深化重构方案。

## 什么是深模块？

小接口藏大实现。更可测试，更 AI 友好。

```
好：深模块 — 简单 API，复杂内部
差：浅模块 — 复杂 API，薄内部
```

## 流程

### 1. 自然探索
记录摩擦点：
- 理解一个概念需要在多个文件间跳转？
- 模块太浅，接口 ≈ 实现？
- 为了可测试性提取纯函数，但 bug 藏在调用方式中？
- 紧耦合模块在接缝处产生集成风险？
- 未测试或难测试的代码？

**遇到的摩擦就是信号。**

### 2. 展示候选
每个机会：
- **涉及模块**：哪些模块
- **耦合原因**：共享类型、调用模式
- **测试影响**：边界测试会替代什么

问用户："探索哪个？"

### 3. 用户选择

### 4. 框定问题
写用户面向的解释：约束、依赖、粗略草图。

### 5. 设计多个接口（并行）
派发 3+ 子代理，每个不同约束：
- Agent 1: "最小化接口 — 最多 1-3 个入口"
- Agent 2: "最大化灵活性"
- Agent 3: "优化最常见调用者"

### 6. 比较和推荐
有主见 — 用户要强判断，不要菜单。

### 7. 创建 GitHub Issue
RFC 用 `gh issue create`。

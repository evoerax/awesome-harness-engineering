# 分类 Issue

调查 bug，找到根因，创建带 TDD 修复计划的 GitHub Issue。

## 流程

### 1. 捕获问题
问一个问题："你看到的问题是什么？"
然后立即开始调查。不追问。

### 2. 探索和诊断
找到：
- **哪里** bug 出现（入口点、UI、API）
- **什么** 代码路径涉及
- **为什么** 失败（根因，不是症状）
- **什么** 相关代码存在

看：
- 相关源文件和依赖
- 已有测试
- 最近变更（`git log`）
- 代码路径中的错误处理
- 正常工作的类似模式

### 3. 确定修复方案
- 修复根因的最小变更
- 受影响的模块/接口
- 需要通过测试验证的行为
- 是回归、缺失功能还是设计缺陷？

### 4. 设计 TDD 修复计划
RED-GREEN 循环有序列表：

```markdown
1. **RED**：写一个测试 [期望行为]
   **GREEN**：[让它通过的最小变更]

2. **RED**：写一个测试 [下一个行为]
   **GREEN**：[让它通过的最小变更]

**REFACTOR**：[全部通过后的清理]
```

### 5. 创建 GitHub Issue
用 `gh issue create`。分享 URL。

## 规则

- 测试通过公共接口，不通过实现
- 一次一个测试，纵向切片
- 描述行为和契约，不描述文件路径
- Issue 应该能经受重大重构

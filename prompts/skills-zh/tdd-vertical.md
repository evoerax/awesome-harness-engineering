# TDD（纵向切片）

集成风格测试 + 纵向切片的测试驱动开发。

## 核心原则

测试通过公共接口验证行为，不验证实现细节。代码可以完全改变，测试不应该变。

## 好测试 vs 差测试

| 好测试 | 差测试 |
|--------|--------|
| 集成风格 | 耦合实现 |
| 练习公共 API | Mock 内部依赖 |
| 描述系统做什么 | 验证怎么做的 |
| 重构后仍通过 | 重构后失败 |

## 反模式：横向切片

```
错（横向）：
  RED:   test1, test2, test3, test4, test5
  GREEN: impl1, impl2, impl3, impl4, impl5

对（纵向）：
  RED→GREEN: test1→impl1
  RED→GREEN: test2→impl2
  RED→GREEN: test3→impl3
```

## 工作流

### 1. 规划
- 确认需要的接口变更
- 确认要测试的行为（优先级）
- 找深模块机会
- 用户批准

### 2. Tracer Bullet
写一个测试 → 失败 → 最少代码 → 通过

### 3. 增量循环
每个剩余行为：RED → GREEN

### 4. 重构
全部测试通过后：
- 消除重复
- 深化模块
- 适当应用 SOLID
- 每步后运行测试

## 每轮检查清单

```
[ ] 测试描述行为，不描述实现
[ ] 测试只用公共接口
[ ] 测试能经受内部重构
[ ] 代码对这个测试是最少的
[ ] 没有投机性功能
```
